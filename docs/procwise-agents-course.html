Why Multiple Agents Instead of One Big AI?

The same reason a hospital has specialists instead of one super-doctor. A cardiologist and a neurologist each focus deeply on their area. ProcWise works the same way — each agent is an expert at one thing: extracting data, comparing prices, or drafting emails. This separation of concerns makes the system reliable, testable, and easy to improve one piece at a time.

The Three-Way Match

One of ProcWise's most powerful features: automatically comparing the purchase order, the original quote, and the final invoice to catch discrepancies.

CODE 
def three_way_match_by_po(self, po_id, tolerance_pct=2.0):
    query = """
    MATCH (p:PurchaseOrder {po_id: $po_id})
      -[:CONTAINS_PO_LINE]->(pl:POLineItem)
      -[:ORDERS_ITEM]->(i:Item)
    OPTIONAL MATCH (ql:QuoteLineItem)-[:QUOTES_ITEM]->(i)
    OPTIONAL MATCH (il:InvoiceLineItem)-[:INVOICES_ITEM]->(i)
    RETURN i.item_description AS item,
      pl.unit_price AS po_price,
      ql.unit_price AS quote_price,
      il.unit_price AS invoice_price"""
PLAIN ENGLISH

Define a function that compares three documents for one purchase order. Allow a 2% price difference before flagging.

This is a graph database query (not SQL). Start by finding the purchase order...

Follow the relationship to find each line item in the PO...

Then find the actual item being ordered.

Now look for the original quote for that same item (if one exists)...

And the invoice for that same item (if one exists)...

Return the item name and all three prices so we can compare them side by side.

Every agent in ProcWise shares the same "brain" — the Cognitive Layer. Think of it like air traffic control at an airport — routing requests, maintaining state, and preventing collisions.

🧠

Working Memory

Redis-backed scratchpad with three scopes: workflow (1 hour), session (24 hours), and long-term (permanent).

🔎

Retrieval Engine

Fuses three search methods (graph, vector, keyword) and re-ranks results for maximum relevance.

🔌

LLM Router

Routes tasks to local or cloud AI models based on complexity. Simple tasks stay local; hard ones go to Claude.

✅

Verification Engine

Validates outputs against schemas and confidence thresholds before they leave the system.

📈

Learning Loop

Captures workflow outcomes to continuously improve agent performance over time.

Working Memory: The Agent's Scratchpad

Each agent can store and retrieve information during a workflow. Data expires automatically based on its scope — like sticky notes that dissolve after a set time.

CODE 
class MemoryScope(Enum):
    WORKFLOW = "workflow"     # 1 hour TTL
    SESSION  = "session"      # 24 hour TTL
    LONG_TERM = "long_term"   # permanent

async def store(self, workflow_id, entry):
    redis_key = self._make_key(
        entry.scope, workflow_id, entry.key)
    ttl = self._ttl_for_scope(entry.scope)
    await self._redis.set(
        redis_key, json.dumps(entry.to_dict()),
        **({"ex": ttl} if ttl else {}))
PLAIN ENGLISH

Define three levels of memory, each lasting a different amount of time:

Workflow: lasts 1 hour — for data relevant only to the current task

Session: lasts 24 hours — for data that might be needed across related tasks

Long-term: permanent — for learned patterns and historical insights

 

To save a memory: build a unique key from the scope + workflow ID + data name...

Look up how long this scope's data should last...

Store it in Redis (a fast in-memory database) with an automatic expiration timer.

LLM Router: Choosing the Right AI

Not every task needs a powerful (and expensive) cloud AI. The LLM Router picks the best model for each job.

CODE 
def route(self, task_type):
    if task_type.value in self._cloud_tasks \
       and self._enable_cloud \
       and self._cloud_api_key:
        return self._cloud_provider, self._cloud_model
    local_model = self._local_map.get(
        task_type.value, self._local_fallback)
    return "ollama", local_model
PLAIN ENGLISH

Given a task type (like "extraction" or "negotiation_strategy"), decide which AI model to use:

If this task type needs cloud-level intelligence...

AND the cloud option is turned on...

AND we have an API key for it — then use the cloud model (Claude).

Otherwise, find the right local model for this task type...

(Fall back to a default local model if no specific one is configured)

Use the local model via Ollama — faster, free, and keeps data private.

extraction💻 Phi-4 (local) — fast pattern matching for pulling data from documents
classification💻 Phi-4 (local) — categorizing documents and items
summarization💻 Mistral (local) — condensing long text into key points
complex_reasoning☁️ Claude (cloud) — multi-step logic and analysis
negotiation_strategy☁️ Claude (cloud) — strategic thinking and recommendation
verification☁️ Claude (cloud) — checking other agents' work for accuracy
💡
The Escalation Pattern

Notice the fallback logic: try local first, escalate to cloud if it fails. This is a common pattern called graceful degradation. If the local AI model can't handle a task, the system automatically tries the more powerful cloud model instead of failing. It's like calling a specialist when the general practitioner can't figure out the diagnosis.

The Knowledge Graph

How ProcWise remembers everything — suppliers, relationships, and patterns

Onboarding + Developers

Three Ways to Find Information

Imagine a library with three different finding systems. ProcWise combines all three and picks the best results from each:

🔸

Knowledge Graph (Neo4j)

Like a card catalog that shows relationships. "This supplier has 5 quotes linked to 3 POs and delivered to 2 locations." Finds connections.

🔬

Vector Search (Qdrant)

Like a "books similar to this" recommendation. Finds documents with similar meaning even if they use different words.

📜

Keyword Search (BM25)

Like a full-text search. Finds exact words and phrases. Best when you know exactly what you're looking for.

Weighted Fusion: Combining All Three

The Retrieval Engine doesn't just pick one search method — it runs all three, weights the results, and then re-ranks them using a specialized AI model.

CODE 
async def retrieve(self, query, collection):
    kg = await self._retrieve_kg(query, ctx)
    vec = await self._retrieve_vector(query, col)
    bm25 = await self._retrieve_bm25(query, col)
    # Weight: KG 30%, Vector 40%, BM25 30%
    for r in kg:  r.score *= 0.3
    for r in vec: r.score *= 0.4
    for r in bm25: r.score *= 0.3
    weighted = kg + vec + bm25
    if self._reranker:
        scores = self._reranker.predict(
            [[query, r.content] for r in weighted])
    return sorted(weighted)[:top_k]
PLAIN ENGLISH

Given a search query, search using all three methods at once:

1. Ask the knowledge graph (finds relationships)

2. Ask the vector database (finds similar meaning)

3. Ask the keyword index (finds exact words)

Weight each source: vectors count slightly more (40%) because meaning-based search is usually most relevant for procurement queries.

Combine all results into one list...

Then use a specialized re-ranking AI model to score each result against the original query for final ordering.

Return only the top results (default: 10).

📚
Why Not Just Use One Search Method?

Each method has blind spots. Keyword search misses synonyms ("fastener" vs "bolt"). Vector search can return loosely related results. Graph queries only find known relationships. By combining all three and then re-ranking with AI, ProcWise gets dramatically better results than any single method alone — like asking a librarian, a search engine, and a subject expert, then letting a judge pick the best answers.

class DiscrepancyState(TypedDict):
    workflow_id: str
    extracted_docs: List[Dict]
    processing_issues: List[Dict]
    mismatches: List[Dict]
    summary: Dict

PLAIN ENGLISH

Define the "form" for discrepancy detection. Every step reads and writes to this same form:

workflow_id — unique ID to track this run

extracted_docs — the documents we're checking

processing_issues — problems found during extraction

mismatches — price/quantity discrepancies found

summary — final report of findings

errors — anything that went wrong

status — where we are in the process

Steps 2-4: Nodes, Edges, Compile

Each step is a pure function (a "node"). Edges define the order. Conditional edges handle branching.

CODE 
graph = StateGraph(DiscrepancyState)

graph.add_node("validate_input", validate_input)
graph.add_node("process_issues", process_issues)
graph.add_node("validate_documents", validate_docs)
graph.add_node("persist_mismatches", persist)
graph.add_node("format_output", format_output)

graph.set_entry_point("validate_input")
graph.add_conditional_edges(
    "validate_input", should_continue,
    {"process_issues": "process_issues",
     "format_output": "format_output"})
graph.add_edge("process_issues", "validate_documents")
graph.add_edge("validate_documents", "persist_mismatches")
graph.add_edge("persist_mismatches", "format_output")
graph.add_edge("format_output", END)

return graph.compile()
PLAIN ENGLISH

Create a new workflow graph using our DiscrepancyState form.

 

Register each processing step as a "node" with a name and function:

1. Validate the input data is good

2. Process any known issues from extraction

3. Compare documents against the database

4. Save any mismatches we found

5. Format the final output report

 

Start at "validate_input". Then branch: if validation passes, go to "process_issues". If it fails, skip straight to "format_output" (to report the error).

After that, it's a straight line: process → validate → persist → format → done.

 

Compile the graph into a runnable workflow.

💡
Why Graphs Beat Classes

The old agents were 1,748-line classes that mixed validation, business logic, LLM calls, and database operations together. A bug in one area could break everything. LangGraph workflows split each concern into a small, testable node function. You can test "validate_input" alone, swap "persist_mismatches" for a mock, or add a new step without touching existing code. This is the Open-Closed Principle in action.

Feature Flags: The Kill Switch

The entire new architecture can be toggled on or off with a single feature flag.

ENABLE_COGNITIVE_LAYER=trueActivates the new LangGraph workflows, cognitive layer, and unified retrieval. Set to false to fall back to legacy agents.
ENABLE_CLOUD_LLM=trueAllows the LLM Router to escalate to Claude for complex tasks. Set to false to keep everything running on local models only.

💡
The Adapter Pattern: Backward Compatibility

Each workflow has an adapter that makes it look like the old agent class to the rest of the system. This means the API endpoints, the orchestrator, and the frontend don't need to change at all. The new workflows slot in seamlessly behind the same interface. When you're confident the new workflow is solid, you can remove the adapter and update the callers — but there's no rush.
